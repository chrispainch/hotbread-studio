# Wardley Core

## Map Structure

A Wardley Map has four essentials:

- `visual representation`
- `context specific`
- `position` of components relative to an anchor
- `movement` as components evolve

For business maps:

- The anchor is `user needs`.
- The vertical axis is the `value chain`, ordered by visibility to the user.
- The horizontal axis is `evolution`.

Evolution stages:

- `genesis`: rare, uncertain, experimental
- `custom built`: bespoke, still being learned
- `product`: repeatable, increasingly well understood
- `commodity / utility`: standardized, high-volume, operationally efficient

## Mapping Workflow

1. Choose scope.
2. Identify the relevant users.
3. State user needs precisely.
4. List top-level capabilities needed to meet those needs.
5. Expand those capabilities into subcomponents until further detail is out of scope.
6. Place components vertically by value chain visibility.
7. Place components horizontally by evolution.
8. Challenge assumptions about novelty, commodity status, and missing dependencies.

Useful prompts:

- What does the user actually need?
- What components are required to provide that?
- Which components are truly differentiating?
- Which are becoming utility-like?
- Are we mapping what exists, or how we wish to describe it?

## Doctrine

Universal principles emphasized by Wardley:

- `focus on user needs`
- `use a common language`
- `be transparent`
- `challenge assumptions`
- `remove duplication and bias`

Practical interpretation:

- Do not confuse company needs with customer needs.
- Use one shared representation for cross-functional discussion.
- Share the map so others can attack weak assumptions.
- Look for redundant bespoke systems and mislabeled “core” components.

## Climatic Patterns

Core patterns to test against a map:

- `everything evolves`
- `characteristics change` as things evolve
- `no one size fits all` methods or management
- `efficiency enables innovation`
- `higher-order systems create new sources of value`
- `no choice on evolution` when competitors adapt
- `past success breeds inertia`

Questions to ask:

- Which components are likely to move right?
- What changes in user expectations follow from that shift?
- Where are we resisting an inevitable utility or commodity transition?
- What new higher-order opportunities become possible if a component industrializes?

## Inertia And Bias

Common sources:

- sunk cost
- pride in existing products or practices
- incumbent revenue models
- organizational structure
- legacy user expectations

Common mistakes:

- treating a commodity as if it were novel
- using one method across all evolution stages
- assuming current customer language is a reliable guide during transitions
- presenting purpose statements as strategy

## Strategy Use

Mapping does not output the answer. It narrows the field of sensible moves.

Use the map to:

- compare strategic options
- explain why one move is preferable to another
- expose doctrine failures
- identify likely future pressure from evolving components
- design experiments in uncertain spaces

## Output Pattern

When using this skill, prefer this structure:

1. `Scope and users`
2. `Map sketch`
3. `Climatic patterns`
4. `Doctrine assessment`
5. `Strategic plays`
6. `Key uncertainties`
