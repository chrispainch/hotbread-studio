# Problem Framing Template

Use this as the default structure for a framing artifact.

```md
# <Title>

## Summary

<2-4 sentences describing the need, the current gap, why the problem matters now, and the likely organizational upside if solved.>

## Primary User

- User:
- Context:
- Job to be done:

## Need

<What the user needs to accomplish or feel able to do. Keep this independent from any proposed solution.>

## Current Gap

<What is missing today. Describe the current state, workaround, friction, or failure mode.>

## Why This Matters

### Qualitative Pain

- <confusion, friction, support burden, trust cost, operational pain, etc.>

### Quantitative Signals

- <known numbers or directional indicators>
- <if unknown, say what is unknown>

## Better User Outcome If Solved

<Describe how the user's life improves once the need is satisfied.>

## Organizational Outcome If Solved

- <business, product, operational, or strategic benefit>
- <business, product, operational, or strategic benefit>
- Attribution confidence: <high / medium / low and why>

## Current Workflow

<Optional short workflow summary or embedded breadboard.>

## Problem Statements

1. <problem statement>
2. <problem statement>
3. <problem statement>

## Priority Signals

- Pain severity:
- Frequency:
- Reach:
- Confidence:
- Strategic fit:

## Open Questions

- <unknown>
- <assumption that needs validation>

## Recommendation

<Is this worth advancing to solution shaping now? If yes, what should shaping focus on first?>
```
